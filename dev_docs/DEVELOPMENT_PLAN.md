# PhoenixKit Manufacturing — план развития (черновик для обсуждения)

Дата: 2026-07-09. Статус: модуль-заглушка подключён (dashboard в админ-меню под Складом). Ниже — предлагаемая архитектура и этапы, всё открыто для обсуждения.

## 0. Что уже есть (техническая база)

- Форк `timujinne/phoenix_kit_manufacturing` (upstream `BeamLabEU/phoenix_kit_manufacturing`), подключён в andi как path dep — та же схема, что у склада.
- Регистрация через `use PhoenixKit.Module`: ключ `manufacturing`, включение через Settings (`manufacturing_enabled`), пункт меню `priority: 154` в группе `:admin_main` — сразу под Складом (153).
- Роуты генерируются автоматически из `admin_tabs/0` (`live_view:` в Tab) — router.ex в andi не трогаем.
- Пока без таблиц БД и миграций.

## 1. Станки (Machines) — справочник

Первый реальный этап: простой CRUD-справочник по образцу складских справочников.

Предлагаемый минимум полей (поля не финализированы — на обсуждение):
- `name`, `code` (короткий код/инвентарный номер), `status` (active / maintenance / decommissioned)
- `machine_type` — тип/группа (фрезер, станок с ЧПУ, пресс…) — отдельный мини-справочник или enum?
- `location` — привязка к складу/помещению (есть `phoenix_kit_locations`)
- `description`, `metadata` (JSONB — под неясные пока поля, чтобы не плодить миграции)
- soft delete + activity log (как в складе)

Открытые вопросы:
- Нужны ли паспортные данные (мощность, рабочая зона, дата ввода в эксплуатацию, график ТО)?
- Станок = ресурс с календарём загрузки (для планирования) или пока просто карточка?

## 2. Документ производства (Production Order)

По аналогии с документами склада (internal_orders / goods_issues / goods_receipts):

- Шапка: номер, дата, статус (draft → in_progress → done → cancelled), станок(и), ответственный, срок.
- Строки: **вход** (материалы/заготовки — позиции каталога, количество) и **выход** (готовая продукция, количество). Фактически упрощённая спецификация (BOM).
- `source_refs` JSONB — та же развязка, что в складе: документ производства может ссылаться на заказ / подзаказ / проект, не зная их таблиц.
- Позже: операции/этапы внутри документа (маршрутная карта), время по операциям.

Вопрос: нужен ли отдельный справочник «Спецификации (BOM)» (шаблон: из чего делается изделие), из которого документ производства заполняется одной кнопкой? Предлагаю в v2, не сразу.

## 3. Связь со складом (передача в производство)

Склад уже спроектирован под это — контракт `source_kinds` (`PhoenixKitWarehouse.SourceKinds`): хост регистрирует в конфиге виды источников с колбэками search/resolve/build_lines, и складские документы (Internal Orders, Goods Issues) умеют ссылаться на них и строить строки.

Схема потока:
1. **Материалы в производство**: производственный документ регистрируется как source kind `production_order` → со склада создаётся Goods Issue (расход) «в производство», строки строятся из входных строк производственного документа (`build_lines`).
2. **Готовая продукция на склад**: по завершении производства — Goods Receipt (приход) с ссылкой на тот же документ.
3. Остатки/резервы считает склад (stock_ledger / committed_quantities) — производство своих остатков не ведёт, только ссылается.

Это ничего не требует менять в складском модуле — регистрация делается в конфиге andi (как уже сделано для order/sub_order через `Andi.Warehouse.Integration`).

## 4. Интеграции с другими модулями

- **Dashboards**: контракт `phoenix_kit_widgets/0` (плоские мапы, без зависимости от phoenix_kit_dashboards). Виджеты: «производственные заказы по статусам», «загрузка станков», «просроченные». Образец — HelloWidget в hello_world.
- **Сотрудники (phoenix_kit_staff)**: ответственный за документ, операторы станков/операций (employments/departments уже есть). Ссылки — через uuid + resolve, без жёсткой зависимости.
- **Проекты (phoenix_kit_projects)**: производственный документ ← проект (source_refs), у проектов уже есть Gantt/календарь — потенциально показывать производственные заказы на таймлайне проекта. Механику связи предлагаю обсудить после появления справочника станков.
- **Каталог (phoenix_kit_catalogue)**: строки документа производства = позиции каталога (как в складе).
- **Права**: `permission_metadata/0` уже заявлен — при появлении разделов добавить отдельные permissions (view/manage), как в складе.

## 5. Предлагаемые этапы

1. **v0.1 (сделано)** — заглушка dashboard + меню + инфраструктура модуля.
2. **v0.2 — Станки**: таблица + CRUD + activity log; поля минимальные, `metadata` JSONB под остальное. Появляется первая миграция (см. вопрос про миграции ниже).
3. **v0.3 — Документ производства**: шапка + строки вход/выход, статусы, ссылки source_refs.
4. **v0.4 — Связка со складом**: source kind `production_order`, кнопки «выдать материалы» / «оприходовать продукцию».
5. **v0.5 — Виджеты dashboards + сотрудники/проекты**.
6. Дальше: BOM-шаблоны, маршрутные карты/операции, планирование загрузки станков.

## 6. Технические вопросы на обсуждение

1. **Миграции**: у склада таблицы живут в versioned-миграциях ядра phoenix_kit, своих миграций в репо модуля нет. Для manufacturing так же (таблицы добавляются в ядро, PR в phoenix_kit) или свой `migration_module/0` внутри модуля? От этого зависит процесс релизов.
2. **Поля станков** — см. §1.
3. **Нумерация документов** — своя серия (MFG-0001) или общий механизм нумерации складских документов?
4. **Меню**: пока один пункт Dashboard; при появлении станков/документов — подпункты (parent: :manufacturing), как у склада.
